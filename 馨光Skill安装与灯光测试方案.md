# 馨光 Skill 安装与灯光测试方案

本文是第二阶段本地草稿。第一阶段安装流程、安装脚本和页面 4 步主流程保持不变。

## 第二阶段目标

第二阶段按研发原始设计收口：小龙虾通过馨光 Skill 控制馨光设备，用户直接用自然语言描述灯光效果。

第二阶段只保留：

1. 安装馨光 Skill。
2. 启动馨光本地灯控服务。
3. 如果有多个米家家庭，让用户选择家庭。
4. 小龙虾读取该家庭下的设备列表。
5. 用户直接说灯光需求。
6. 馨光 Skill 执行灯光效果。

第二阶段不并入第一阶段默认安装流程，也不加入首页正式入口，除非后续另行确认。

## 必须保留：米家家庭选择

多家庭选择是第二阶段必须保留的安全步骤。

1. 如果只有一个家庭，可以继续。
2. 如果有多个家庭，必须展示家庭列表。
3. 用户选择家庭后再继续。
4. 不允许自动使用第一个家庭。

页面文案统一为：

```text
如果你的米家账号下有多个家庭，请选择馨光设备所在的家庭。
```

## 用户主流程

用户直接描述灯光需求，不需要额外选择灯光设备。

推荐表达：

```text
完成前面步骤后，可以继续安装馨光 Skill。
安装完成后，如果你的米家账号下有多个家庭，请选择馨光设备所在的家庭。
之后你可以直接告诉小龙虾想要的灯光效果，例如“客厅设计一个马尔代夫灯光效果”，小龙虾会通过馨光 Skill 控制馨光设备。
```

用户可以直接输入：

```text
客厅设计一个马尔代夫灯光效果。
二楼客厅来一个森林晨光。
卧室做一个适合睡前放松的灯光。
全屋灯带做一个朋友聚会氛围。
保存当前灯光效果到快照 3。
```

## 内部处理原则

设备识别交给馨光 Skill，不写进用户主流程。

内部可以保留：

1. 读取当前家庭设备列表。
2. 根据用户语句识别房间和灯光需求。
3. 在当前家庭范围内选择合适的馨光灯光设备执行。
4. 找不到设备时，提示检查米家账号、家庭和设备在线状态。

不要把产品选择、设备编号、设备能力判断写成用户必填步骤。

## 单次测试规则

真实灯光测试仍保留单次测试规则：

1. 一次只发送一个灯光请求。
2. 请求发出后等待用户现场确认。
3. 不自动重试。
4. 不自动关灯。
5. 不切换颜色继续验证。
6. 不做反向控制判定。
7. 不控制其它设备。

如果现场灯光已经变化，但接口返回状态不一致，应记录为：

```text
PHYSICAL_SUCCESS_API_FALSE
```

如果出现连续变色、关灯再亮或多条命令叠加，应记录为：

```text
UNSTABLE_MULTIPLE_COMMANDS
```

## 研发修复项

以下问题记录为研发修复项，不进入用户主流程：

1. 自动修 token。
2. 自动 patch 二进制。
3. 自动诊断接口返回状态。
4. 自动反复发控制命令。
5. 自动判断所有设备边界。
6. 接口返回状态不一致后的自动重试。
7. 复杂设备识别解释。

## 给小龙虾的安装指令

正式指令见：

```text
docs/发给小龙虾的馨光Skill安装指令.txt
```

执行目标：

1. 不重复执行第一阶段安装流程。
2. 不修改第一阶段安装脚本。
3. 下载固定版本的第二阶段安装脚本。
4. 安装馨光 Skill。
5. 启动馨光本地灯控服务。
6. 检查米家家庭列表。
7. 如果需要，让用户选择米家家庭。
8. 读取所选家庭下的设备列表。
9. 安装完成后，提示用户直接用自然语言描述灯光需求。

## 验收标准

第二阶段通过验收，需要满足：

1. 馨光 Skill 已安装。
2. 馨光本地灯控服务已启动。
3. 多家庭场景不会自动使用第一个家庭。
4. 用户选择家庭后，小龙虾能读取该家庭下的设备列表。
5. 用户可以用自然语言提出灯光需求。
6. 小龙虾能在当前家庭范围内调用馨光 Skill。
7. 馨光设备有实际灯光变化。
